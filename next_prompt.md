# Техническое Задание: Hotfix LLM Collapse (Server)

<context>
Анализ логов (`pm2 logs`) с продакшен сервера выявил причину "Коллапса LLM".
Next.js (в `apps/web/src/app/api/chat/route.ts`) выдает ошибку `UND_ERR_HEADERS_TIMEOUT` (Headers Timeout Error).
**Суть:** Встроенный в Node.js 18+ клиент `fetch` (базируется на `undici`) имеет жесткий хардкод-лимит ожидания заголовков: ровно 5 минут (300000 ms). Наш `AbortSignal.timeout(900_000)` на это НЕ влияет. Так как Gemini 3.1 Thinking может "думать" 4-6 минут до отправки ответа, Next.js Proxy принудительно рвет соединение, и Ассистент на фронте просто зависает или выдает `fetch failed`.
</context>

## Задачи для выполнения Кодером (Срочно)

1. **Замена `fetch` в Next.js Proxy на отказоустойчивый клиент**
   - В файле `C:\project\VITOGRAPH\apps\web\src\app\api\chat\route.ts` полностью удали использование нативного `fetch` для перенаправления запросов.
   - Используй библиотеку `axios`. Установи её: `cd apps/web && npm install axios`.
   - Перепиши POST роут: сделай запрос через `axios.post(backendUrl, json, { headers, timeout: 900000 })`. axios корректно поддерживает кастомные таймауты до ответа.
   - Сделай то же самое для метода `DELETE`.

2. **Фикс JWT Expired (PGRST303) в Бэкенде (Опционально-важно)**
   - В логах бэкенда: `[handleChat] Error inserting user msg: ... JWT expired`.
   - Проверь метод сохранения сообщений в историю чата в бэкенде. Ошибки вставки из-за просроченного токена не должны крашить сам функционал ответа пользователю (используй `try/catch` с мягким логированием ошибки, если не может сохранить лог, чтобы LLM всё равно ответила).

3. (Для Архитектора/Саши - не для кодера) После выкладки фикса, пользователи не будут падать по HeadersTimeout, и будут мирно дожидаться долгого ответа от Gemini.
